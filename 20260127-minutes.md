 - Investigate providing a PAM module which implements 'irods' authentication
   - Yes, possible...
   - Post-5.1.0, to demonstrate a pam-only installation
     - Would allow removal of the native authentication
     - Would use a server-to-server forever token
   - Would demonstrate a password-less iRODS catalog
     - Just tokens in our table
     - thumbs-up/thumbs-down decisions decided by PAM stack
   - Could even be offloaded to a separate space, not even iRODS table
 - Claudio, currently...
   - SURF has multiple services today
   - Users use more than one in their normal workflows
   - Web-world, can have OIDC for SSO, seamless to move between services
   - But most users are not web-only
   - Sometimes iRODS is behind web, but sometimes only API / CLI
   - How can we unify, provide a transparent login for all users
   - Security requirements demand user-specific permissions
     - Requires real authentication
     - Nearly all have pushed to using token-only protocol
       - Token exchange, then temporary access with timeout
       - Most are bouncing to web login, then copy/paste token to API/CLI
       - Delegates application to work on behalf of a user via token
     - Pam interactive can now be the receiver of access tokens
   - What if you want iRODS to be the coordinating service… calling other services
     - Would have to receive a token, and then connect to another service
       - Which would require to get another token to the next service
       - Then use it
     - Or iRODS calls a hook or service to get a new token on behalf of a user
       - But who provides this?  iRODS?  Or admins of a service
     - Token exchange protocol
       - As user, get a new token for the next service
       - RFC 8693 - https://datatracker.ietf.org/doc/html/rfc8693 
   - Example...
     - iRODS to publish data into a repository, dataset/metadata/etc.
       - Possibly dataverse specifically
     - iRODS pushes to an API, but the repository has its own users
     - Can iRODS push into that repository AS that user
     - Suggests there a unique/user-mapping from iRODS user to API user
     - Rather than having a service account to connect and act with privileges
       - Security audits are NOT liking this style of deployment
     - Maybe this is 2-3 'functions' or rules that we ship
       - Get_8693_token()
       - Use_8693_token()
       - Or do_8693_stuff(user, token, endpoint, payload)
